01

What it is

htmla は Claude Code が「HTML 形式で何かを作って」と頼まれたときに、 castle で定義された ivory + clay デザイントークン（serif 見出し / system sans 本文 / JetBrainsMono Nerd Font）を 使って 単一ファイルで完結する HTML を出力するためのスキル。

  • 外部 CSS / Web フォント / CDN を使わない。受け手のマシンで素のブラウザで開けば動く。
  • 毎回 :root ブロックを HTML 冒頭に再宣言する。design-system.html はコピペ元として機能する設計。
  • 色は CSS 変数経由でのみ参照。リテラル hex を直書きしない。
  • SVG はインライン。外部画像に依存させない。
02

発火させる 3 つのコツ

Claude Code のスキルは description のキーワードマッチで自動発火する。 意図せず別のスキルが動いたり、htmla が動かなかったりするときは、以下のコツで誘導する。

Tip 01 · keyword 「HTMLで」「artifact」「デザインを揃えて」

これらの語彙のいずれかを含めると自動発火する。逆に避けたい場合は「Markdown で」を明示する。

Tip 02 · who · what 「誰に何を伝えるか」を書く

「モックアップ作って」より「新規ユーザー向けに 3 案 横並び比較」の方が design-system のどのコンポーネントを選ぶかの判断材料になる。

Tip 03 · output 出力先と single-file を明示

tmp/spec.html に single-file で と書くと、場所決めで迷わず、:root のコピペ運用も自然に維持される。

03

用途別プロンプト集

コピペで使えるテンプレ。@ でファイル参照、@path 部分は適宜置き換えて使う。

A仕様書 / 実装プラン

新しい通知センター機能の実装プランを HTML で書いて。
データフロー図、主要な画面のモックアップ、API のリクエスト/レスポンス例、
リスクと回避策を 1 ファイルにまとめて tmp/notification-spec.html に保存。
design-system.html を参照してデザインを揃えて。
@src/auth/ を読んで、現状の認証フローを HTML 図解で説明して。
SVG でシーケンス図、コードの該当箇所を inline で引用、
gotchas を最後にコールアウトでまとめて。tmp/auth-explainer.html に。

B探索 / 比較

オンボーディング画面の方向性を 6 案、HTML で並べて比較できるようにして。
それぞれ tone（formal / playful / minimal）と density（low / high）を
ばらして、各案の trade-off を mono で添えて。
このバグの修正方法を 3 通り考えて HTML で並べて。
各案の影響範囲、テスト戦略、ロールバック容易性を pros/cons カードで。
推奨案には clay の "Recommended" pill をつけて。

Cレポート / 振り返り

直近 1 週間の git log を読んで、変更を機能別にカテゴライズした
週次レポートを HTML で。KPI（commits / PRs / 行数）を上に、
カテゴリごとのカード一覧をその下に、最後に来週やることを serif-list で。
このコードベースのテスト戦略を HTML で説明して。
テストの種類（unit / integration / e2e）ごとの位置付けを SVG で図解、
各 layer の代表的なテストファイルへのリンクを cards で並べて。

DPR レビュー / コード理解

このブランチの差分を HTML で解説。
変更概要、影響を受けるモジュール、注意すべきレビューポイントを
セクション分け。重要な diff を pre.code で抜粋、
気になる箇所には callout-warn でコメント。
@useAuth.ts を読んで、この hook の挙動を HTML で図解して。
状態遷移図（SVG）、各 state の意味（table）、
よくある誤用パターン（callout-note）を含める。

Eデザイン / プロトタイプ

チェックアウトボタンのアニメーションを 4 パターン、HTML で試作。
各パターンに sliders（duration / easing / scale）をつけて、
気に入った値を「copy as JSON」できるボタンも欲しい。
ダッシュボードのカードレイアウト案を、grid の breakpoints を
切り替えられる HTML で。tablet / desktop / wide の 3 段階で
プレビューできるように。

Fカスタムエディタ / 一回限りツール

この Linear のチケット 30 件を re-prioritize したい。
Now / Next / Later / Cut の 4 列に drag-and-drop できる
HTML editor を作って。最後に「copy as markdown」ボタンで
並び替え結果と各バケットの 1 行 rationale を出力。
このフィーチャーフラグ設定の編集 UI を HTML で作って。
依存関係（A が ON なら B も ON）を warn で表示、
変更分だけを diff として「copy diff」できるボタンつきで。

G概念の解説

プロンプトキャッシングの仕組みを HTML explainer で。
読者は中級エンジニア前提。流れを SVG で図解、
コード例を 2-3 個、最後に "gotchas" セクション。tmp/prompt-cache-explainer.html に。
04

表現の調整

プロンプトに足す一言で、出力のトーンや構造を調整できる。

やりたいこと足す一言
自動発火させたいHTMLで / HTML artifact / デザインを揃えて
確実に発火させたい/htmla を頭につけて明示起動
文体を硬くしたい「editorial tone で」「日経ビジネス風に」
文体を軽くしたい「カジュアルなトーンで」「内輪向けの memo として」
単一ファイル維持single-file で / 外部 CSS なしで
出力先を固定tmp/<name>.html に保存して
図を多めにSVG で図解を最低 2 つ / flow を絵で
数字を強調KPI で開始 / dashboard 風に
比較したい<n> 案を横並びで / pros/cons を mono で添えて
インタラクティブtabs で章立て / sliders で調整できる
05

使わない場合（Markdown のままでいい）

逆に Markdown のままでよいケース。HTML 化は overkill になりやすいので、 短文や標準フォーマットの成果物では Markdown を選ぶ。

  • ~/.zshrc 等への 1〜2 行追加レベルの説明
  • README / CHANGELOG / リリースノート — Markdown が標準フォーマットの成果物
  • 1 問 1 答の short answer
  • Claude Code 内の short conversation で完結する内容（HTML 化したら共有手段が必要になる）
  • diff レビューを頻繁にする想定のドキュメント（HTML diff はノイジー）
06

閲覧方法

castle にコミットされた HTML を見るには 3 通り。共有相手や状況に応じて使い分ける。

①

htmlpreview.github.io 経由（GitHub Web からそのまま見たいとき）

URL を https://htmlpreview.github.io/? で prefix する。第三者サービス経由なので機密情報を含む HTML には使わない。

https://htmlpreview.github.io/?https://github.com/branch10480/castle/blob/main/docs/htmla-usage.html
②

ローカルで open（自分で見るとき推奨）

JetBrainsMono Nerd Font が効くので、フォント込みでデザインが完璧に再現される。castle が homeshick link 済みなら以下で開ける。

open ~/.homesick/repos/castle/docs/htmla-usage.html
③

raw 直リンク（GitHub Web で .html を開く）

これはソースコード表示になるため rendered されない。GitHub Web で .html をクリックしても view-source 状態になるので、共有時は ① か ② を使う。

＋

将来オプション · GitHub Pages 化

HTML が増えてきたら branch10480.github.io/castle/ として公開する選択肢もある。短くきれいな URL になる代わり、castle 全体の公開範囲を判断する必要がある。
