# PR レビュー&修正ループ

PR `$ARGUMENTS` に対して、`code-review-expert` または `swift-code-reviewer` サブエージェントによるレビューと修正を、品質基準を満たすまで繰り返す。**差分はリモートではなくローカルの git から取得する**ため、修正のたびに push する必要はない。

## 起動方法

```
/review-loop [<PR番号 or PR URL>]
```

引数が省略された場合は、現在のブランチに紐づく PR を `gh pr view --json number` で特定する。特定できなければユーザーに確認する。

## 設計方針

- **品質は高く保つ**: BLOCKER（バグ・セキュリティ・破壊的変更）と MAJOR（設計欠陥・性能劣化・API 誤用）は必ず潰す
- **nit は許容**: 命名・フォーマット・コメント文言レベルの好みでは LGTM を保留しない
- **収束性を担保**: 2 周目以降は差分レビュー化して指摘の自然増殖を防ぐ
- **ハード上限**: 4 周で強制終了して人間にエスカレーション
- **push しない**: 差分はローカル `git diff` / `git show` で取得し、push のタイミングはユーザーに委ねる（CI / TestFlight トリガーなど外部影響を避けるため）

## 実行手順

### 0. 前準備

1. **PR 番号を確定**する（引数 or `gh pr view --json number`）— 識別・PR コメント連携用途
2. **ベースブランチを確定**する: `BASE=$(gh pr view $PR --json baseRefName -q .baseRefName)`（通常 `main`）
3. **ベースとの分岐点を確定**する: `BASE_REF=$(git merge-base origin/$BASE HEAD)`
4. **全差分を取得**: `git diff $BASE_REF..HEAD`（リモート PR ではなくローカル commit を見る）
5. **差分ファイル一覧を取得**: `git diff --name-only $BASE_REF..HEAD`
6. **レビュー履歴用の状態を保持**: `iteration=1`, `previous_findings=[]`, `fix_commit_count=0`

### 0.5. レビュアー選定

差分ファイル一覧を解析し、起動するレビュアーを決定する:

| 差分の中身 | 起動エージェント |
|---|---|
| 以下のいずれかを含む（マッチはケースセンシティブ）: `*.swift` / `*.xcodeproj` / `*.xcworkspace` / `Package.swift` / `Package.resolved` / `*.pbxproj` / `*.storyboard` / `*.xib` / `*.xcprivacy` / `*.entitlements` / `*.xcconfig` / `Info.plist` | `swift-code-reviewer` |
| Swift 系ファイルが含まれない | `code-review-expert` |
| Swift 系と非 Swift 系が混在 | **両方を並列起動** し、結果をマージ（同一ファイル・同一行の指摘は dedup） |

選定結果は周回間で固定する（差分が同じであれば毎周同じレビュアーを使う）。

### 1. レビュー（毎周）

選定したエージェントを `Agent` ツールで起動する（混在ケースは 1 メッセージ内で複数 Agent 呼び出しを並列発火）。プロンプトには以下を**必ず含める**:

```
PR #<PR番号> のブランチをレビューしてください。
**差分はローカル commit を直接参照すること。リモートには push されていない可能性があります。**

## レビュー対象差分の取得方法

- 1 周目: ベースからの全差分
  - 差分: `git diff $BASE_REF..HEAD`（$BASE_REF = `git merge-base origin/$BASE HEAD`）
  - ファイル一覧: `git diff --name-only $BASE_REF..HEAD`
- 2 周目以降: 直近の修正コミット群のみ
  - 差分: `git diff HEAD~<fix_commit_count>..HEAD`
  - 単一コミットの場合は `git show HEAD` でも可
  - **`gh pr diff` は使用禁止**（push 漏れで古い差分を見るリスクあり）

## 対象範囲

- 1 周目: ブランチ全体をフルレビュー
- 2 周目以降: 前回指摘の修正箇所、およびその修正による副作用が及ぶ範囲のみ
  （前回の指摘リスト: <previous_findings>）
  ※ 全体の再レビューはしないこと

## 重要度分類（必須）

出力は以下のフォーマットに従うこと。

### [BLOCKER] 必ず修正が必要
- バグ / セキュリティ脆弱性 / 正しくない動作 / データ破壊 / 破壊的変更

### [MAJOR] 修正推奨
- 設計上の欠陥 / パフォーマンス劣化 / API 誤用 / クリティカルパスのテスト不足

### [MINOR] 任意（修正しなくても LGTM）
- リファクタの余地 / 非クリティカルなテスト追加余地

### [NIT] 出力禁止
- 命名の好み / コメント文言 / フォーマット
- このカテゴリは絶対に出力に含めないこと

## 判定（必須）

最後に以下のいずれかを明記:
- 判定: LGTM   （BLOCKER と MAJOR が 0 件のとき）
- 判定: NEEDS_WORK （BLOCKER または MAJOR が 1 件以上のとき）

## 注意

- レビュー対象の実コードを必ず読み、誤指摘がないか確認してから出力すること
- 推測ベースの指摘は禁止
- 2 周目以降は新規論点を持ち込まないこと（修正適切性のみ判定）
```

### 2. 結果の解析と分岐

レビュー結果をパースして次のいずれかに進む:

| 状態 | アクション |
|------|------------|
| 判定 = LGTM | **終了**: ユーザーに完了報告（修正サマリ + 残存 MINOR + push 案内を提示） |
| 判定 = NEEDS_WORK かつ 指摘が前回と同一 | **エスカレーション終了**: 「2 周連続で同じ指摘が残存。設計見直しが必要」と報告 |
| 判定 = NEEDS_WORK かつ iteration == 4 | **上限到達終了**: 残存指摘を一覧で提示して終了 |
| 判定 = NEEDS_WORK かつ それ以外 | **修正フェーズへ**（手順 3） |

**「同一」の判定ルール**: 各指摘を `(ファイルパス, カテゴリ, 内容のキーフレーズ)` のタプルで比較し、**BLOCKER + MAJOR の集合が前回と同じ**であれば「同一」と判定する。MINOR は判定対象外。文言が完全一致でなくても、同じ箇所・同じ性質の指摘ならば同一とみなす。

### 3. 修正

- BLOCKER と MAJOR の全指摘に対応する
- MINOR は明らかに低コストなものだけ対応（ループを長引かせない）
- 修正完了後、変更を **commit する**（**push はしない、ユーザーに任せる**）
  - 1 周あたり 1 コミットを推奨（複数の fix を 1 commit に集約 → `git diff HEAD~1..HEAD` で次周のレビュースコープが明確になる）
- `previous_findings` を今回の BLOCKER + MAJOR リストに更新
- `fix_commit_count += 1`
- `iteration += 1`
- 手順 1 に戻る

## 終了時の報告フォーマット

```markdown
## レビューループ結果

- 結果: LGTM / エスカレーション / 上限到達
- 反復回数: N / 4
- 修正済み BLOCKER: X 件
- 修正済み MAJOR: Y 件
- 残存 MINOR: Z 件（任意対応）
- 残存 BLOCKER/MAJOR: あれば一覧

## 修正コミット（ローカル — push されていない可能性あり）
- <commit hash>: <subject>
...

## 次のアクション
- リモートへ反映する場合: `git push`
- マージは GitHub 上で実施（必要なら追加レビュー再依頼）
```

## 注意事項

- **差分取得はローカル diff ベース**（`git diff` / `git show`）を使う
  - 理由: 1 周ごとに push を強制すると CI / TestFlight など外部への影響が出るため、push タイミングはユーザーに委ねる
  - `gh pr diff` は使用しない（push 漏れで古い差分を見るリスク）
  - PR 番号は識別と PR コメント連携用途のみ
- ループ完了後（LGTM・エスカレーション・上限到達のいずれでも）、ユーザーに「`git push` するか、PR を更新するか」の判断を委ねる
- GitHub メタデータ（PR 番号・ベースブランチ名など）の参照は `gh` コマンドを使う（差分そのものはローカル `git`）
- 機密情報（API キー等）が修正コミットに混入しないか毎周確認する
- ユーザーの中断（Ctrl-C 等）はいつでも受け付ける
- レビュー対象の実コードを必ず参照し、誤指摘を含めないこと（CLAUDE.md 方針に従う）
