$ARGUMENTSについて詳細なコードレビューを実行します。

## オプション

- `--comment` または `-c`: レビュー結果をPRにコメントとして投稿します
- `--draft` または `-d`: ドラフトコメントとして保存します（投稿せずに下書き保存）

使用例:
```bash
/pr-review 123 --comment       # PR #123をレビューして結果をコメント投稿
/pr-review 456 -c              # 短縮形でのコメント投稿
/pr-review 789 --draft         # ドラフトとして保存
/pr-review 101                 # レビューのみ（コメント投稿なし）
```

## 事前準備とエージェント活用

まず、以下を確認・実行してください：

1. **エージェントの確認**: `.claude/agents/` ディレクトリの内容を確認し、利用可能なエージェントを読み込む
2. **プロジェクト構造把握**: GrepツールとReadツールを活用してコードベース全体構造を理解
3. **レビューガイドライン確認**: `.github/review-bot.yml` または類似のレビュー要件ファイルを確認

## レビュー実行手順

### Phase 1: オプション解析とPR情報収集

#### 1.1 オプション解析
まず引数を解析してオプションフラグを確認：

```bash
# 引数解析（PR番号と--commentオプションの検出）
ARGS="$ARGUMENTS"
PR_NUMBER=""
COMMENT_MODE=""

# オプション解析
if [[ "$ARGS" =~ --comment|-c ]]; then
    COMMENT_MODE="comment"
    ARGS=$(echo "$ARGS" | sed -E 's/--comment|-c//g')
elif [[ "$ARGS" =~ --draft|-d ]]; then
    COMMENT_MODE="draft"
    ARGS=$(echo "$ARGS" | sed -E 's/--draft|-d//g')
fi

# PR番号抽出（空白を除去）
PR_NUMBER=$(echo "$ARGS" | sed 's/[^0-9]//g')
echo "PR番号: $PR_NUMBER"
echo "コメントモード: ${COMMENT_MODE:-なし}"
```

#### 1.2 PR情報収集
以下のコマンドを**すべて実行**してPR情報を収集：

```bash
# PRの基本情報（タイトル、説明、変更ファイル一覧、URL）
gh pr view $ARGUMENTS --json title,body,files,url

# 詳細な差分情報
gh pr diff $ARGUMENTS

# 既存のコメント確認
gh pr view $ARGUMENTS --comments

# ファイル別コメント（レビューコメント）
gh api repos/{owner}/{repo}/pulls/$ARGUMENTS/comments | jq '.[] | {file: .path, line: .line, user: .user.login, body: .body, created_at: .created_at}'
```

### Phase 2: コンテキスト分析とコードベース理解

1. **変更の背景分析**: PRの説明とコミットメッセージから変更の意図を理解
2. **影響範囲調査**: 変更されたファイルに依存する他のファイルをGrepツールで検索
3. **アーキテクチャ整合性**: 既存のコードパターンとの一貫性を確認

### Phase 3: 詳細コードレビュー

各変更ファイルについて以下を実行：

#### 3.1 構文・スタイル・パターン分析
- **コードスタイル**: 既存のコーディング規約との整合性
- **命名規則**: 変数名、関数名、クラス名の適切性
- **デザインパターン**: アーキテクチャパターンの遵守

#### 3.2 品質・セキュリティ分析
- **コードスメル**: 重複コード、長大なメソッド、複雑な条件分岐
- **セキュリティ**: 入力検証、認証・認可、データ漏洩リスク
- **パフォーマンス**: 無駄なAPI呼び出し、メモリリーク、処理効率

#### 3.3 機能・ロジック分析
- **ビジネスロジック**: 要件との整合性、エッジケースの考慮
- **エラーハンドリング**: 例外処理の適切性
- **データ整合性**: データベースやAPIとの整合性

### Phase 4: 依存関係とテスト分析

1. **依存関係影響調査**: Grepツールでimpact analysis実行
2. **テスト要件判定**: 変更内容に応じたテストの必要性を既存テストパターンから判断
3. **リグレッションリスク**: 既存機能への影響評価

### Phase 5: 総合評価とフィードバック生成

### Phase 6: コメント投稿（オプション）

レビュー完了後、`--comment` または `--draft` オプションが指定されている場合の処理：

#### 6.1 コメント投稿処理
```bash
# コメントモードに応じた処理
if [[ "$COMMENT_MODE" == "comment" ]]; then
    echo "📝 レビュー結果をPR #$PR_NUMBER にコメントとして投稿します..."

    # レビュー結果をコメントとして投稿
    gh pr comment "$PR_NUMBER" --body "$(cat << 'EOF'
# 🔍 コードレビュー結果

[ここにレビュー結果の内容を挿入]

---
*このレビューは Claude Code の PR Review 機能により自動生成されました*
EOF
)"

    echo "✅ コメント投稿完了"

elif [[ "$COMMENT_MODE" == "draft" ]]; then
    echo "📝 レビュー結果をドラフトとして保存します..."

    # ドラフトレビューとして保存
    gh pr review "$PR_NUMBER" --comment --body "$(cat << 'EOF'
# 🔍 コードレビュー結果（ドラフト）

[ここにレビュー結果の内容を挿入]

---
*このレビューは Claude Code の PR Review 機能により自動生成されました*
EOF
)"

    echo "✅ ドラフトレビュー保存完了"

else
    echo "📋 レビュー完了（コメント投稿なし）"
fi
```

#### 6.2 コメント内容の構造化
投稿するコメントは以下の構造で自動フォーマット：

```markdown
# 🔍 コードレビュー結果

## 📋 PR概要
- **目的**: [変更の目的]
- **影響範囲**: [影響するコンポーネント]
- **リスクレベル**: [Low/Medium/High]

## 🚨 必須修正事項
[must] ラベルの項目をリスト化

## 💡 改善提案
[want] ラベルの項目をリスト化

## ❓ 確認事項
[ask] ラベルの項目をリスト化

## 📈 総合評価
- ✅ **承認** / ⚠️ **要修正** / ❌ **要大幅修正**
- 主要懸念: [懸念事項がある場合]

---
*🤖 このレビューは Claude Code により生成されました*
```

## レビューコメント作成ガイドライン

### コメント分類とラベル
- **[must]**: 必須修正（バグ、セキュリティ、破壊的変更）
- **[want]**: 推奨修正（パフォーマンス、保守性向上）
- **[ask]**: 質問・確認事項（意図確認、代替案提案）
- **[nits]**: 軽微な指摘（スタイル、命名、コメント）
- **[info]**: 情報提供（参考資料、ベストプラクティス）
- **[imo]**: 個人的見解（主観的な改善提案）

### コメント構造
```
[ラベル] ファイル名:行番号
簡潔な指摘内容（最大3行）
理由・根拠
```

### 出力セクション構成

#### 1. 📋 PR概要とコンテキスト
- 変更の背景・目的
- 技術的アプローチの妥当性
- アーキテクチャへの影響

#### 2. 📁 ファイル別分析
各ファイルの変更内容と評価

#### 3. 🚨 必須修正事項
セキュリティ、バグ、破壊的変更など

#### 4. 💡 改善提案
パフォーマンス、保守性、拡張性の観点から

#### 5. ❓ 確認事項
意図不明な実装や代替案に関する質問

#### 6. 🧪 テスト要件
追加すべきテストの種類と優先度

#### 7. 📈 総合評価
- 承認可否の判定
- 主要な懸念事項
- 次のアクションアイテム

## 実行時の注意事項

- **並行処理**: 情報収集は効率化のため並行実行
- **コード検索活用**: Grep/Globツールでコード理解を促進
- **コンテキスト重視**: 単一ファイルでなく全体設計への影響を考慮
- **建設的フィードバック**: 問題指摘と改善案をセットで提供
- **優先度明確化**: 修正の緊急度と重要度を明示
- **です・ます調**: 丁寧で建設的なトーン

## コメント投稿に関する注意事項

- **デフォルト動作**: `--comment` オプション未指定時はレビュー結果の表示のみ
- **投稿前確認**: `--comment` 指定時は投稿内容をユーザーに確認してから実行
- **認証要件**: `gh` コマンドでGitHubにログイン済みであることが必要
- **権限確認**: 対象リポジトリへのwrite権限またはreview権限が必要
- **ドラフト機能**: `--draft` オプションでレビューを下書き保存（後で編集・投稿可能）

## ⚠️ 制限事項

**禁止されているコマンド:**
- `git merge` - ローカルマージ
- `git pull` - リモートから取得してマージ
- `gh pr merge` - PRマージ
- `gh repo merge` - リポジトリレベルのマージ

これらのマージ関連コマンドは安全性の観点から使用できません。レビュー機能はコードの品質確認とフィードバック提供のみを行い、マージの判断・実行は人間が行うべきです。

**重要**: コメント投稿オプション使用時は必ずユーザーの明示的な承認を得てから実行してください。マージは手動で行ってください。
