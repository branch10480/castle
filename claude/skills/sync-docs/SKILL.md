# sync-docs

## 手順

### 1. 現在のプロジェクト構成を取得

Glob や find でプロジェクトのソースファイル一覧を取得する。`.git/`, `DerivedData/`, `node_modules/`, `.build/`, `.claude/` は除外する。

### 2. 既存ドキュメントを読み込む

- `CLAUDE.md` を読む
- `AGENTS.md` を読む（Codex / 他エージェント共通の指示。検索順: `./AGENTS.md` → `.codex/AGENTS.md` → castle なら `home/.codex/AGENTS.md`。存在しないリポジトリではスキップ）
- `README.md` を読む
- `docs/specification.md` を読む（開発フェーズ進捗・設計方針の確認用）
- データモデルの実装ファイルがあれば読む（プロパティ名・型の確認用）
- 直近の `git log --oneline -10` を確認し、最近の変更を把握する

### 3. 差分を特定して修正

**CLAUDE.md:**
- ディレクトリ構成ツリーが実際のファイル/フォルダと一致しているか（順序はファイルシステムの実態に従う）
- 開発フェーズのチェックボックスが進捗と一致しているか
- 新規ファイル/ディレクトリの追加漏れ、削除済みファイルの残存がないか
- 技術方針・規約が実装と矛盾していないか
- 「注意事項」セクションに以下の漏れがないか:
  - 新規サービス/メソッドの実装パターン（例: キャッシュ戦略、メモリ最適化手法）
  - Pro/無料の切り分けに関する情報
  - 新しいデータフローや依存関係

**AGENTS.md（存在する場合）:**
- CLAUDE.md と共通する事項（ディレクトリ構成・命名規約・コマンド一覧・「やってはいけない事」）が乖離していないか
- CLAUDE.md だけ更新して AGENTS.md が取り残されていないか（双方を真実の源とする運用なので片寄せ NG）
- Codex 固有の事項（pet-assets、`.codex/config.toml` の machine-local 注意、`AGENTS.md` 自身の symlink ルールなど）は AGENTS.md にだけ書き、CLAUDE.md に重複させない
- 同じファクトを両方に書く場合は**表現を揃える**（数値・パス・キーワードが食い違うとレビュー時に「どっちが正？」となる）

**README.md:**
- データモデルのコード例が実装と一致しているか（プロパティ名、型、enum）
- 開発フェーズの進捗が最新か
- 技術スタックに変更がないか

**docs/specification.md:**
- 開発フェーズのロードマップが CLAUDE.md と一致しているか
- 実装済み機能のステータスが正しいか

### 4. 修正ルール

- 実装を正とし、ドキュメントを合わせる
- **CLAUDE.md** = Claude Code 向け詳細（ディレクトリツリー、注意事項、コーディング規約）
- **AGENTS.md** = Codex / 他エージェント向けの同等指示。CLAUDE.md と共通事項は鏡像（同じ事実は同じ表現）、エージェント固有の事項は片方にのみ
- **README.md** = ユーザー/新規開発者向けサマリー（データモデル例、技術スタック、セットアップ手順）
- **docs/specification.md** = 設計仕様・フェーズ計画（CLAUDE.md の開発フェーズと整合性を保つ）
- 完了タスクは `[x]` / `~~取り消し線~~` にする
- フォーマットは既存スタイルを踏襲する
- 変更なしなら「差分なし、最新です」と報告して終了

### 5. 変更サマリーを報告

修正箇所を箇条書きでユーザーに報告する。
