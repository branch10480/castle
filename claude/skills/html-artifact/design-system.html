Code

3 styles

Code blocks sit on the warm --g100 tint (not pure gray) so they stay in the warm ivory family. Inline code uses the same tint with a small radius. The token-row style is for design-system listings or config diffs.

  return `<aside style="border-left: 4px solid ${accent}">`;

Callouts

3 tones

Left-border callouts are the loudest layout move in this system. Reserve them for things the reader must not skim past. The italic serif "ico" is intentional — it reads as a pull-quote, not a UI alert.

Tabs

stateless · JS-light

Tabs are CSS + a 10-line script. Underline animates with the clay accent. Keep tabs to 3-5 — more than that, prefer a TOC pill row at the top of the document.

Why we picked this approach. Short narrative — this should be readable in 30 seconds.

Code-level detail. Acceptable to inline a short pre.code block.

What could break. Be specific — "rate limit on third-party API" beats "performance".

09

Lists

2 patterns

Two list flavors: the serif numbered list (for steps / decisions) and the dot list (for short scannable items). Avoid the browser default — they're indistinguishable from markdown rendering.

Serif numbered

  1. Brief Claude. Specify the artifact type (spec / report / mockup / explainer) and the audience.
  2. Point at this design system. Tell Claude to read design-system.html first.
  3. Single-file output. Inline the :root block; no external CSS, no build step.
  4. Open in browser. Eyeball it. Iterate verbally.

Dot list

  • Use --clay for accents only — never for body text or large fills.
  • Italics on the serif read as quiet emphasis; pair with clay for loud emphasis.
  • Keep tabs to 3-5; longer needs a TOC pill row instead.
  • Cards always include a mono-styled .file footer.
